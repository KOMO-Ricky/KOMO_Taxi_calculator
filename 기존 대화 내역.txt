구글 스프레드시트의 CSV export는 CORS 정책 때문에 브라우저에서 직접 fetch()로 호출하면 막힙니다. 공개 설정을 해도 동일한 문제가 발생해요. Google Apps Script를 프록시로 사용하는 방식으로 바꿔야 합니다.

해결 방법: Google Apps Script API 배포
1단계 — 스프레드시트에서 Apps Script 열기

해당 스프레드시트에서: 확장 프로그램 → Apps Script

2단계 — 아래 코드를 붙여넣기

javascript
function doGet(e) {
  const ss = SpreadsheetApp.openById('16tTQilanjKsumRLmSHrbeBghgdMNF2p9FjemZq-9Qco');
  const sheet = ss.getSheetByName('자금계산기 DB');
  
  const data = {
    licensePrice: sheet.getRange('B3').getValue(),
    commission:   sheet.getRange('B6').getValue(),
    carPrice:     sheet.getRange('F3:F6').getValues().flat(),
    meterWork:    sheet.getRange('G3:G6').getValues().flat(),
    taxFee:       sheet.getRange('H3:H6').getValues().flat(),
    insurance:    sheet.getRange('I3:I6').getValues().flat(),
    bbFHD:        sheet.getRange('L3:L6').getValues().flat(),
    bbQHD:        sheet.getRange('M3:M6').getValues().flat(),
    bb4K:         sheet.getRange('N3:N6').getValues().flat(),
    fixed:        sheet.getRange('Q3:Q7').getValues().flat()
  };
  
  return ContentService
    .createTextOutput(JSON.stringify(data))
    .setMimeType(ContentService.MimeType.JSON);
}

회사에서 사용할 DB와 웹사이트인데 보안 문제가 발생할 수 있지 않을까?

좋은 질문입니다. 보안 측면에서 정리해 드릴게요.

현재 구조의 보안 이슈
Apps Script 방식의 핵심 문제는 "모든 사용자에게 공개"로 배포하면 URL을 아는 누구나 DB 데이터에 접근 가능하다는 점입니다. 회사 업무용이라면 아래 두 가지를 반드시 고려해야 합니다.

보안 수준별 선택지
✅ 옵션 A — Apps Script + 접근 제한 (권장, 비용 없음)
Apps Script 코드에 비밀 토큰 검증을 추가합니다.

javascript
function doGet(e) {
  // 요청에 올바른 토큰이 없으면 차단
  const SECRET = 'your-secret-key-here'; // 임의 문자열 설정
  if (e.parameter.token !== SECRET) {
    return ContentService.createTextOutput(JSON.stringify({ error: 'unauthorized' }))
      .setMimeType(ContentService.MimeType.JSON);
  }
  // ... 나머지 데이터 반환 코드
}
계산기 HTML에서는 이 토큰을 포함해서 호출하게 됩니다. 단, HTML 소스를 보면 토큰이 노출되므로 완전한 보안은 아닙니다. 내부 직원만 쓰는 환경이라면 충분할 수 있습니다.

✅ 옵션 B — 구글 계정 로그인 필수 (가장 안전, 비용 없음)
Apps Script 배포 시 액세스를 "Google 계정으로 로그인한 사용자" 또는 **"조직 내 사용자만"**으로 제한합니다.

배포 설정 → 액세스 권한: 조직 내 사용자 (Google Workspace) 또는 로그인된 사용자

이 경우 회사 Google 계정으로 로그인한 사람만 데이터를 볼 수 있습니다. 외부 유출 위험이 없습니다.

✅ 옵션 C — 아예 외부 통신 없이 HTML에 하드코딩 (가장 단순)
시세나 비용이 자주 바뀌지 않는다면, 스프레드시트 연동 없이 값을 HTML에 직접 넣고, 변경이 필요할 때 파일만 교체하는 방식입니다. 보안 이슈가 없고 구조도 단순합니다.

정리
방식	보안 수준	편의성	비용
Apps Script (공개)	❌ 낮음	✅ 편함	무료
Apps Script + 토큰	🔶 중간	✅ 편함	무료
Apps Script + 구글 로그인	✅ 높음	🔶 로그인 필요	무료
HTML 하드코딩	✅ 높음	🔶 수동 업데이트	무료

사내 인트라넷 전용인가요, 외부 인터넷에도 공개되는 페이지인가요?
아임웹 사이트가 비공개(로그인 필요) 페이지인가요, 공개 페이지인가요?
정확히는 회사에서 고객들에게 마케팅용으로 배포할 웹사이트야. 그런데 저 스프레드시트DB가 공개 설정되면 고객들이 DB에 접근할 수 있는 거 아니야?

정확히 이해하셨습니다. 결론부터 말씀드리면:

고객이 DB에 "직접 접근"하는 건 아니지만, URL을 알면 누구나 데이터를 볼 수 있습니다.

구글 스프레드시트를 "링크가 있는 모든 사용자 - 뷰어"로 설정하면, URL만 있으면 누구든 스프레드시트 자체를 열람할 수 있어요. 마케팅용 공개 웹사이트에 올라간 HTML 소스를 보면 시트 ID가 그대로 노출되기 때문에 실질적으로 공개 데이터와 다름없습니다.

마케팅 배포용에 적합한 구조
고객에게 배포하는 웹사이트라면 Apps Script를 백엔드로 쓰되, 스프레드시트는 비공개로 유지하는 게 정답입니다.

고객 브라우저 → Apps Script 웹앱 URL → (비공개) 스프레드시트
이 구조의 핵심은 이렇습니다.

스프레드시트는 비공개 유지 → 고객이 URL을 알아도 접근 불가
Apps Script는 스프레드시트 소유자 권한으로 실행 → 비공개 시트도 읽을 수 있음
고객에게는 Apps Script URL만 노출 → 데이터 구조/원본 시트 비공개
즉, 스프레드시트 공개 설정을 다시 비공개로 되돌리셔도 Apps Script를 통하면 정상 작동합니다.

URL 주소명 추천을 위해 이 계산기가 배포될 사이트(tlxc.co.kr)와 서비스 성격을 고려해서 제안하겠습니다.

핵심 원칙부터 말하면, URL은 짧고, 기억하기 쉽고, 내용을 예측할 수 있게 만드는 게 좋습니다. 이 서비스는 "개인택시 준비자금 계산기"이므로 그 핵심 키워드를 살리는 방향이 자연스럽습니다.

추천 (기존 도메인 tlxc.co.kr 하위 경로)
가장 무난하고 실용적인 방식입니다.

tlxc.co.kr/calc — 짧고 직관적, 가장 추천
tlxc.co.kr/calculator — 명확하지만 조금 김
tlxc.co.kr/cost — 비용 계산 뉘앙스
tlxc.co.kr/fund — 자금 계산 뉘앙스
한글 서비스라면 이렇게도 가능합니다 (한글 URL도 요즘 문제없이 작동합니다):

tlxc.co.kr/계산기
tlxc.co.kr/준비자금
조금 더 구체적으로
계산기가 여러 개 생길 가능성을 염두에 둔다면 카테고리를 두는 게 좋습니다.

tlxc.co.kr/calc/fund (준비자금)
tlxc.co.kr/calc/loan (대출)
이렇게 하면 나중에 다른 계산기를 추가할 때 구조가 깔끔하게 확장됩니다.

별도 서브도메인을 쓴다면
계산기를 독립 서비스처럼 키우고 싶다면:

calc.tlxc.co.kr
cost.tlxc.co.kr
몇 가지 실무 팁을 덧붙이면, 하이픈이나 언더스코어가 많은 긴 주소(gaein-taxi-jagem-calculator)는 피하는 게 좋고, 대소문자는 섞지 말고 전부 소문자로 통일하는 게 안전합니다. 그리고 아임웹은 게시판/페이지별로 URL 슬러그를 직접 지정할 수 있으니, 페이지 설정에서 원하는 경로를 넣으면 됩니다.

한 가지 확인하면 더 정확히 추천할 수 있는데, 이 계산기를 아임웹 사이트 안의 한 페이지로 둘 생각인가요, 아니면 독립된 별도 주소로 접근하게 할 생각인가요? 그리고 앞으로 대출 계산기 같은 다른 계산기도 추가할 계획이 있는지도 알려주시면 확장까지 고려한 구조를 제안하겠습니다.
